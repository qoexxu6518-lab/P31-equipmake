# 独立游戏 AI 辅助开发标准

> 本标准用于指导 AI 从零开始制作 2D 独立游戏，覆盖从原型到 Steam 上架的全流程。
> 技术栈：HTML/CSS/JS + Canvas/PixiJS + Electron 打包。

---

## 一、项目定位

**产出物**：一个可在浏览器中运行、可打包成桌面端的 2D 独立游戏。

**适用范围**：
- 肉鸽/割草（Vampire Survivors 类）
- 回合制 RPG / 策略
- 塔防 / 放置经营
- 卡牌构筑（杀戮尖塔类）
- 平台跳跃 / 动作（中等复杂度）
- 文字冒险 / 视觉小说
- 弹幕射击

**不适用**：3D 游戏、重度物理模拟、竞技级高帧率要求。

## 三、游戏主循环标准结构

```javascript
// main.js
const canvas = document.getElementById('gameCanvas');
const ctx = canvas.getContext('2d');

let lastTime = 0;

function gameLoop(timestamp) {
  const dt = (timestamp - lastTime) / 1000; // 秒
  lastTime = timestamp;

  update(dt);   // 逻辑更新（固定时间步长可选）
  render(ctx);  // 渲染

  requestAnimationFrame(gameLoop);
}

function update(dt) {
  // 1. 处理输入
  // 2. 更新玩家
  // 3. 更新敌人/AI
  // 4. 更新投射物
  // 5. 碰撞检测
  // 6. 处理掉落物/经验
  // 7. 更新粒子特效
  // 8. 更新 UI 状态
  // 9. 检查游戏结束条件
}

function render(ctx) {
  // 1. 清屏
  // 2. 绘制地图/背景（含相机偏移）
  // 3. 绘制实体（敌人 → 掉落物 → 玩家 → 投射物）
  // 4. 绘制粒子特效
  // 5. 绘制 HUD（血条、分数、时间）
}

requestAnimationFrame(gameLoop);
```

## 四、核心系统设计模板

### 4.1 场景状态机

```
MENU → PLAYING → PAUSED → PLAYING → GAME_OVER → MENU
                                   → UPGRADE_PICK → PLAYING
```

每个场景有独立的 `update(dt)` 和 `render(ctx)`，场景切换通过 `SceneManager.switchTo('playing')`。

### 4.2 实体管理

```javascript
// 对象池模式，避免频繁 GC
const enemyPool = [];
function spawnEnemy(x, y, type) {
  let e = enemyPool.find(e => !e.active);
  if (!e) { e = new Enemy(); enemyPool.push(e); }
  e.init(x, y, type);
  return e;
}
```

### 4.3 碰撞检测

- 少量实体（<100）：直接 AABB 两两检测
- 大量实体（>100）：**四叉树 / 空间网格** 分区优化
- AI 在实体数量增长后自动重构为分区检测

### 4.4 相机系统

```javascript
const camera = { x: 0, y: 0 };
// 跟随玩家，居中显示
camera.x = player.x - canvas.width / 2;
camera.y = player.y - canvas.height / 2;
// 绘制时所有世界坐标减去 camera 偏移
ctx.drawImage(sprite, entity.x - camera.x, entity.y - camera.y);
```

### 4.5 存档系统

```javascript
function saveGame(data) {
  localStorage.setItem('gameSave', JSON.stringify(data));
}
function loadGame() {
  const raw = localStorage.getItem('gameSave');
  return raw ? JSON.parse(raw) : null;
}
// Electron 版改用 fs 写入文件
```

## 五、开发阶段（渐进式推进）

### 阶段 0：参考游戏拆解与系统规划
**目标**：明确"保留什么、替换什么、新增什么、删除什么"

这是大部分项目的起手方式。以"参考 A 游戏，把 X 系统改成 Y"为起点：

- [ ] AI 检索参考游戏，输出**系统拆解清单**
- [ ] 用户确认每个系统的处理方式（保留 / 替换 / 新增 / 删除）
- [ ] AI 输出**最终系统规划表**（见下方模板）
- [ ] 确认核心循环：玩家做什么 → 得到什么 → 为什么继续玩

#### 系统拆解清单模板

AI 检索参考游戏后，按以下格式输出：

```
参考游戏：《Vampire Survivors》

系统拆解：
1. 移动系统 — WASD 俯视角自由移动，无跳跃
2. 武器系统 — 自动攻击，最多6把武器叠加，可进化
3. 敌人系统 — 四面涌来，波次递增，有精英/Boss
4. 成长系统 — 拾取经验 → 升级 → 随机3选1（武器/被动）
5. 被动道具系统 — 最多6个槽位，提供属性加成
6. 进化系统 — 武器+指定被动 → 进化为终极武器
7. Meta进度 — 金币 → 永久属性强化 + 角色解锁
8. 关卡系统 — 多张地图，限时30分钟生存
```

#### 系统规划表模板

用户确认后，AI 输出最终规划：

```
┌──────────────┬──────────┬──────────────────────────┐
│ 系统          │ 处理方式  │ 说明                      │
├──────────────┼──────────┼──────────────────────────┤
│ 移动系统      │ 保留     │ 照搬 WASD 俯视角          │
│ 武器系统      │ 替换     │ 改为卡牌构筑：抽卡打出释放 │
│ 敌人系统      │ 保留     │ 照搬波次递增              │
│ 成长系统      │ 替换     │ 改为卡牌获取+牌组构筑      │
│ 被动道具系统  │ 保留     │ 照搬被动槽位              │
│ 进化系统      │ 删除     │ 卡牌构筑下不需要          │
│ Meta进度      │ 保留     │ 照搬金币永久强化          │
│ 关卡系统      │ 保留     │ 照搬限时生存              │
│ 卡牌商店      │ 新增     │ 战前组牌，战后购卡        │
└──────────────┴──────────┴──────────────────────────┘
```

> **检查点**：系统规划表确认后才进入阶段 1。如果参考游戏较冷门，用户需提供视频/截图/文字描述，AI 据此完成拆解。

### 阶段 1：最小可玩原型（核心循环）
**目标**：验证核心玩法是否有趣

- [ ] 画布初始化 + 游戏主循环
- [ ] 玩家角色：移动（WASD/方向键）
- [ ] 基础敌人：生成、朝玩家移动
- [ ] 一种武器/攻击方式
- [ ] 碰撞检测：攻击命中敌人、敌人接触玩家
- [ ] 生命值：玩家受伤、敌人死亡
- [ ] 游戏结束判定

> **检查点**：能玩 30 秒，有基本的操作 → 挑战感。不好玩就在这个阶段调整核心机制。

### 阶段 2：成长系统
**目标**：让玩家有变强的体验

- [ ] 经验值/掉落物收集
- [ ] 升级选择（随机 N 选 1）
- [ ] 多种武器/技能
- [ ] 被动能力（攻击力、移速、范围等）
- [ ] 敌人波次/难度递增
- [ ] 数值配置表（config.js 集中管理）

> **检查点**：能玩 5 分钟，有成长感和构筑策略差异。

### 阶段 3：内容填充
**目标**：丰富游戏内容

- [ ] 多种敌人类型（不同行为模式）
- [ ] Boss 战
- [ ] 多角色（不同初始武器/属性）
- [ ] 多关卡/地图
- [ ] 道具/宝箱系统
- [ ] 解锁系统（永久进度）

> **检查点**：能玩 15-30 分钟，有重复游玩动力。

### 阶段 4：视听打磨
**目标**：提升手感和沉浸感

- [ ] 精灵动画（走路、攻击、死亡）
- [ ] 粒子特效（击杀爆炸、升级光效、屏幕震动）
- [ ] 伤害数字飘字
- [ ] 音效接入（攻击、受伤、升级、拾取、按钮）
- [ ] BGM（菜单、战斗、Boss）
- [ ] 屏幕震动 / 击中顿帧（Hit Stop）
- [ ] UI 动画（血条平滑变化、弹出面板过渡）

> **检查点**：操作有反馈感，打击感满足。

### 阶段 5：完整游戏流程
**目标**：从启动到退出的完整体验

- [ ] 主菜单（开始、继续、设置、退出）
- [ ] 设置面板（音量、分辨率、按键）
- [ ] 暂停菜单
- [ ] 结算画面（统计数据、评分）
- [ ] 存档/读档
- [ ] 新手引导（可选）
- [ ] 多语言支持（可选）

> **检查点**：完整流程无断裂，感觉像一个成品。

### 阶段 6：桌面端打包
**目标**：打包成可发布的桌面应用

- [ ] Electron 项目初始化
- [ ] 窗口配置（尺寸、全屏、图标）
- [ ] 存档迁移到文件系统
- [ ] electron-builder 配置（Windows/Mac/Linux）
- [ ] 构建测试

### 阶段 7：Steam 上架（可选）
**目标**：发布到 Steam 平台

- [ ] Steamworks 开发者注册
- [ ] steamworks.js 集成
- [ ] Steam 成就系统
- [ ] Steam 云存档
- [ ] Steam 商店页面配置
- [ ] 上传构建 + 审核

## 六、性能优化清单

当游戏出现卡顿时按优先级排查：

| 优先级 | 优化项 | 做法 |
|--------|--------|------|
| 1 | 对象池 | 敌人、投射物、粒子复用，不频繁 new/GC |
| 2 | 碰撞分区 | 四叉树或空间网格，减少检测次数 |
| 3 | 离屏剔除 | 不在画面内的实体跳过渲染 |
| 4 | 批量绘制 | 合并同类 drawImage 调用 |
| 5 | 降低更新频率 | 远处敌人 AI 降频更新（每 3 帧一次） |
| 6 | Web Worker | 碰撞/寻路等重计算移到后台线程 |
| 7 | OffscreenCanvas | 静态背景预渲染到离屏画布 |

## 七、数值平衡规范

```javascript
// config.js - 集中管理所有可调参数
const CONFIG = {
  player: {
    baseHP: 100,
    baseMoveSpeed: 200,     // 像素/秒
    invincibleTime: 0.5,    // 受伤无敌帧（秒）
    pickupRange: 60,
  },
  enemy: {
    spawnInterval: 1.0,      // 秒
    spawnRateGrowth: 0.95,   // 每波乘数（越来越快）
    baseHP: 10,
    hpGrowthPerWave: 1.15,   // 每波血量乘数
    baseDamage: 10,
    moveSpeed: 80,
  },
  exp: {
    baseToLevel: 10,
    growthFactor: 1.2,       // 每级所需经验乘数
  },
  // ...所有数值都在这里，不散落在代码各处
};
```

> **原则**：所有影响游戏体验的数字都必须在 config 中定义，方便统一调整。AI 修改平衡性时只改 config，不动逻辑代码。

## 九、阶段交付与检查原则

| 原则 | 说明 |
|------|------|
| **阶段 0 必须先完成** | 系统规划表确认前不写代码，避免推倒重来 |
| **先能玩再好玩** | 阶段 1 用色块/占位图就行，不要在原型阶段纠结美术 |
| **保留的系统忠实还原** | "保留"意味着照搬参考游戏的该系统逻辑，不自作主张修改 |
| **替换的系统优先实现** | 替换系统是游戏差异化的核心，阶段 1-2 必须验证 |
| **每个阶段必须可运行** | 不允许出现"改完一堆但跑不起来"的情况 |
| **频繁验证** | 每完成一个子任务就在浏览器中测试 |
| **数值集中管理** | 所有平衡参数放 config.js，不散落在逻辑代码中 |
| **AI 不猜测玩法** | 核心玩法取决于你的设计，AI 只负责实现和建议 |
| **渐进式复杂度** | 单文件 → 多文件 → 模块化，代码结构随需要自然演进 |
| **美术最后接** | 先用简单图形验证手感，确认好玩后再替换正式素材 |
